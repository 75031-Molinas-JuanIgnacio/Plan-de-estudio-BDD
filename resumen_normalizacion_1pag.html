Fundamentos
Equivalencias clave
  • Relación = TABLA
  • Tupla = FILA
  • Atributo = COLUMNA
  • Cardinalidad = nro. de filas
  • Grado = nro. de columnas
  • Dominio = tipo de dato (conjunto atómico)
Reglas de Claves
  • PK: unicidad + minimalidad
  • PK nunca puede ser NULL (integridad de entidad)
  • Natural: usa datos reales del dominio
  • Artificial: secuencia numérica (ID)
  • Usar artificial solo cuando no hay opción
  • FK: debe concordar con PK referenciada
  • FK puede ser NULL o valor válido
  • FK puede referenciar la misma tabla
DEPENDENCIA FUNCIONAL
A → B
Cada valor de A tiene exactamente 1 valor de B.
A = determinante | B = dependiente
EJEMPLOS:
NroEmpl → Nombre
NroEmpl → NroSuc → DirSuc (TRANSITIVA ⚠️)
Tipos de DF a detectar
  • DF Completa: B depende de TODO A (PK compuesta)
  • DF Parcial: B depende solo de PARTE de la PK → viola 2FN
  • DF Transitiva: A→B y B→C → viola 3FN
Formas Normales
SN
Sin Normalizar
Datos mezclados, redundancia, anomalías
⬇ aplicar 1FN
1
1FN — Dominios Simples
✓ Cada celda tiene UN SOLO valor atómico
✓ No hay grupos repetidos ni listas
Violación: "tel: 123, 456" en una celda
⬇ aplicar 2FN
2
2FN — Sin DF Parciales
✓ En 1FN
✓ Todos los attr. no-clave dependen de TODA la PK
✓ Solo importa si PK es compuesta
Violación: desc_art depende solo de cod_art (no de toda la PK)
⬇ aplicar 3FN
3
3FN — Sin DF Transitivas
✓ En 2FN
✓ Ningún attr. no-clave depende transitivamente de la PK
Violación: NroEmpl→NroSuc→DirSuc
⬇ (opcional)
B
BCNF — Todo determinante es clave candidata
Más estricta que 3FN. Solo para relaciones con múltiples claves candidatas compuestas con atributo común.
Anomalías (problema de NO normalizar)
Inserción
Para agregar nueva sucursal sin empleados → NULL en PK (ILEGAL)
Borrado
Borrar el último empleado de una sucursal → perdés datos de la sucursal
Modificación
Cambiar dirección de sucursal → hay que actualizarlo en TODAS las filas
Principios de Diseño
Entendimiento del Problema Entender ≠ Interpretar. El documento manda. Sin supuestos extra.
Correcta Dimensión Ni entidades de más (sin doc.) ni de menos (que falten). Solución acotada.
Elocuencia ESTUDIANTES > EST1. Si alguien más puede leerlo sin explicación, es elocuente.
Nombre Explícito fecha_nacimiento > fecha1 > f_nac. Cuanto más detallado, mejor.
Clasificaciones Contenidas Valores de tipo: sexo, tipo_doc → tabla de referencia. El modelo se explica solo.
PK Natural Usar atributos reales del dominio. Control de integridad que las artificiales no dan.
PK Artificial Cuando no hay opción (ej: tabla SEXOS — no hay clave natural obvia).
Misma PK = ¿misma entidad? Si 2 entidades tienen misma PK → posible fusión. Relación 1:1 = probable error.
Valor Repetitivo Texto con 2+ valores que se repite sistemáticamente → tabla de referencia (excepto apellidos/nombres).
Igualdad Estructural ESTUDIANTES y DOCENTES con misma estructura → PERSONAS + TIPOS_FUNCIONES (opcional, no obligatorio).
Pertenencia Natural Atributo va donde produce menos esfuerzo de recuperación y mantenimiento.
Atrib. Calculables Evitar redundancia, pero analizar antes de eliminar. Considerar costo de recalcular.
